# Architecture Decision Records (ADR)

Каталог архитектурных и процессных решений в формате [MADR](https://adr.github.io/madr/).

## Зачем

ADR — это короткий документ, фиксирующий **значимое** решение: контекст, рассмотренные варианты, выбранный путь и его последствия. Цель — иметь восстановимый по git-истории след *почему* проект устроен так, как устроен.

## Когда писать ADR

- Архитектурный выбор (язык, фреймворк, паттерн, протокол).
- Введение/смена ключевого инструмента (CI, тест-раннер, ORM, deployment).
- Изменение процессов работы агентов и людей.
- Любое решение, которое будет неочевидно через 3 месяца человеку (или новому агенту), пришедшему в проект.

## Когда НЕ писать ADR

- Тривиальные правки кода и багфиксы.
- Изменения, легко обратимые и затрагивающие один файл/модуль.
- Стилистические соглашения (если только это не системный сдвиг).

## Как писать

1. Скопируй [`template.md`](template.md) в `NNNN-<kebab-slug>.md`, где `NNNN` — следующий свободный номер (4 цифры).
2. Заполни секции. Будь краток: ADR — не диссертация.
3. Открой PR с типом `docs`. Заголовок: `docs(repo): PTR-XXX adopt <decision summary>`.

## Жизненный цикл ADR

- `Proposed` — обсуждается.
- `Accepted` — принят и применён.
- `Deprecated` — больше не действует, но без замены.
- `Superseded by ADR-MMMM` — заменён другим решением.

Принятые ADR **не редактируются** (кроме опечаток). Изменение позиции = новый ADR со ссылкой `Supersedes ADR-NNNN`.
