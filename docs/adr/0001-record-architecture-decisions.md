# 0001 — Использовать MADR для записи архитектурных решений

- **Статус:** Accepted
- **Дата:** 2026-05-03
- **Авторы:** @enotbert
- **Связано с:** —

## Контекст

PTR v2 — мультиязыковая монорепа под автономную AI-разработку. Большая часть кода и архитектурных шагов выполняется AI-агентами; человек выполняет роль владельца и финального ревьюера. Чтобы агенты (текущие и будущие) понимали, *почему* система устроена так, как устроена, и не «откатывали» осознанные решения по незнанию, нужен явный, версионируемый журнал архитектурных решений.

## Рассмотренные варианты

- **Вариант A: MADR (Markdown Any Decision Records).** Стандарт де-факто, лёгкий шаблон, хорошо ложится в monorepo, читается людьми и AI без специальных тулов.
- **Вариант B: Ничего, фиксировать решения только в PR-описаниях.** Плюс — нулевой оверхед. Минус — решения теряются в массе PR; в новом контексте агент не найдёт «почему».
- **Вариант C: Полноценные RFC с фазой обсуждения и отдельным процессом.** Плюс — детальность. Минус — слишком тяжело для соло-разработчика и быстрых AI-итераций.
- **Вариант D: ADR в `.ai/adr/`.** Плюс — рядом с правилами агентов. Минус — расходится со стандартом сообщества (`docs/adr/`), хуже читается внешними инструментами.

## Решение

Принимаем **MADR в `docs/adr/`**:

- Файлы: `docs/adr/NNNN-<kebab-slug>.md`, где `NNNN` — четырёхзначный последовательный номер.
- Шаблон: [`docs/adr/template.md`](template.md).
- Индекс и инструкции: [`docs/adr/README.md`](README.md).
- Ссылка на каталог явно прописана в [`AGENTS.md` §6](../../AGENTS.md#6-архитектурные-решения-adr) и в [`.ai/rules/00-overview.md`](../../.ai/rules/00-overview.md).

## Последствия

- **Положительные:**
  - Любое значимое решение восстановимо без чтения всего PR-листа.
  - Новые агенты и люди получают контекст из одного места.
  - Стандартный формат — сторонние инструменты (включая ADR-вьюеры) работают «из коробки».
- **Отрицательные / компромиссы:**
  - Дисциплина: нужно реально писать ADR, а не «забыли — потом».
  - Лёгкая дублирующая нагрузка к описаниям в PR (но PR — про *что*, ADR — про *почему*).
- **Что нужно сделать:**
  - Включить упоминание ADR в `pull_request_template.md` (чек-лист «решение требует ADR? — добавил/нет»).
  - Первый PR, вводящий язык/фреймворк, обязан сопровождаться ADR.
