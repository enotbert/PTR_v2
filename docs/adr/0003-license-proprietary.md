# 0003 — Использовать проприетарную лицензию (All Rights Reserved)

- **Статус:** Accepted
- **Дата:** 2026-05-03
- **Авторы:** @enotbert
- **Связано с:** [LICENSE](../../LICENSE)

## Контекст

Репозиторий публичный (`enotbert/PTR_v2`), но это не open-source проект — это рабочая платформа для собственных продуктов. Без явного `LICENSE` публичный репо без лицензии:

- По дефолту защищён авторским правом (`All rights reserved`), но это неочевидно для случайных читателей.
- GitHub не отображает badge с лицензией, и автоматические инструменты не могут классифицировать проект.
- Не запрещает явно копирование и модификацию — формально, без лицензии любое использование сторонним лицом нарушает копирайт, но граница «что считается использованием» размыта.

Нужен явный документ, фиксирующий правовой режим.

## Рассмотренные варианты

- **Вариант A: MIT.** Максимально либеральная, без copyleft. Допускает использование, модификацию, коммерческое использование, при сохранении copyright notice. Минус — раздаём всё бесплатно, включая бизнес-логику.
- **Вариант B: Apache-2.0.** Аналогично MIT + явный патентный грант. Те же минусы.
- **Вариант C: AGPLv3 / GPL.** Copyleft. Любые производные обязаны быть открытыми под той же лицензией. Защищает от proprietary-форков, но крайне жёсткая для возможных партнёрств.
- **Вариант D: Proprietary / All Rights Reserved.** Явный запрет на использование/модификацию/распространение без письменного согласия. Контроль остаётся у владельца. ✅ выбран.
- **Вариант E: Сделать репо приватным.** Полное скрытие. Рассматривался, но публичный репо удобнее для интеграций (gh/Linear/Cursor cloud agents) и не несёт прямого риска при правильной лицензии.

## Решение

Принимаем **Proprietary / All Rights Reserved**:

- Файл `LICENSE` в корне с полным текстом и copyright `Copyright (c) 2026 enotbert`.
- Контрибуции принимаются только по отдельному письменному соглашению (зафиксировано в самом тексте LICENSE).
- Репозиторий остаётся **публичным** для удобства интеграций и хостинга AI-агентов; публичность не подразумевает разрешения на использование кода.

## Последствия

### Положительные

- Полный контроль над использованием, модификацией и распространением.
- Сохраняется возможность коммерциализации без обязательств перед сторонними лицензиями.
- Чёткий сигнал: репо — не open-source, не «бери и используй».

### Отрицательные / компромиссы

- **Невозможны внешние pull request'ы без отдельного соглашения.** Любой контрибутор (включая случайного) формально не может передать код. Если в будущем понадобится принимать community-контрибуции — потребуется CLA или смена лицензии (новый ADR).
- **GitHub не покажет lib license badge.** GitHub не классифицирует проприетарные лицензии в стандартный список SPDX. Это косметика.
- **Зависимости от open-source кода требуют дисциплины.** Все добавляемые библиотеки должны иметь совместимые лицензии (отсутствие copyleft, кроме случаев, когда они касаются изолированных компонентов). Эту дисциплину обеспечит будущий процесс review зависимостей (см. `.ai/rules/40-code-quality.md` § Зависимости).
