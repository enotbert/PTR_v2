# 70 — Оркестрация: Cursor → Codex CLI

> **Статус:** базовый протокол; точные флаги CLI и модель — placeholder, уточняются отдельным PR.
> **Решение:** Cursor вызывает Codex CLI **как subprocess** через свой Shell-инструмент. Передача задачи фиксируется письменно в `.ai/handoffs/`.

## Когда Cursor делегирует Codex CLI

Делегируем, когда задача:

- Чисто кодинговая (реализация по понятной спецификации).
- Однозначна по объёму и критериям приёмки.
- Не требует параллельных архитектурных решений посреди работы.

Не делегируем (Cursor делает сам):

- Документация и правила (`AGENTS.md`, `.ai/rules/`, ADR, README).
- Архитектурные изменения, проектирование схем данных, дизайн API.
- Триаж багов с непонятной причиной.
- Любая задача, где спецификация ещё не сформулирована.

## Хэндофф

Перед запуском Codex CLI Cursor создаёт файл:

```
.ai/handoffs/PTR-XXX-<kebab-slug>.md
```

### Структура хэндоффа

```markdown
# PTR-XXX — <Title>

## Goal
<Что должно быть сделано — 1–3 абзаца.>

## Context
<Ссылки на релевантный код, документацию, ADR. Что уже существует.>

## Files in scope
- path/to/file1.ext  — <короткое пояснение, что именно изменить>
- path/to/file2.ext  — <…>

## Out of scope
<Чего НЕ трогать. Любые проблемы за пределами скоупа — только зафиксировать в Result, не править.>
```

## Запуск Codex CLI (placeholder)

Базовый шаблон вызова из Cursor (через Shell):

```bash
codex --oss \
  --model <agreed-model> \
  --workdir . \
  --instructions .ai/handoffs/PTR-XXX-<slug>.md \
  --non-interactive
```

Альтернатива (если CLI читает спеку через stdin):

```bash
codex --oss --model <agreed-model> < .ai/handoffs/PTR-XXX-<slug>.md
```

После завершения Cursor:

1. Читает раздел `## Result` в хэндоффе.
2. Прогоняет команды из секции `Commands` ещё раз для независимой валидации.
3. Сравнивает диф с `Files in scope` — расхождения требуют объяснения.
4. Если результат не соответствует критериям — возвращает Codex CLI с уточнённой спекой (новая итерация в том же файле, раздел `## Iteration N`).

## Версионирование хэндоффов

- Хэндоффы коммитятся в репо вместе с кодом задачи: они часть аудита.
- Один файл на задачу. Уточнения и итерации — добавлением секций, не перезаписью.
- После merge задачи хэндофф **остаётся** в истории (не удаляется).

## Что Codex CLI **не должен** делать в рамках вызова

- Выходить за пределы `Files in scope` без явного основания.
- Менять `.ai/rules/`, `AGENTS.md`, `.github/`, lockfiles, файлы с секретами.
- Открывать PR, пушить в `origin`, менять статусы в Linear — это делает оркестратор после валидации.
- Запускать деструктивные shell-команды.

## Передача управления человеку

Если Cursor после нескольких итераций (≥ 2) с Codex CLI не получает приемлемый результат:

1. Перевести задачу в Linear в **Blocked** с описанием причины.
2. Перевести PR (если открыт) в **Draft**.
3. Запросить у человека: уточнить спеку, изменить подход, или взять задачу вручную.
