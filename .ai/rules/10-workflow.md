# 10 — Цикл работы агента

> Полная версия минимального цикла из [`AGENTS.md` §4](../../AGENTS.md#4-минимальный-цикл-работы-агента).
> Этому циклу должен следовать **любой** агент, берущий задачу `PTR-XXX`.

## Шаг 0. Подготовка контекста

Перед началом работы агент:

1. Читает [`AGENTS.md`](../../AGENTS.md) и все `.ai/rules/*.md`, релевантные задаче.
2. Подтягивает свежий `main`:
   ```bash
   git checkout main && git pull --ff-only
   ```
3. Проверяет, что рабочее дерево чистое (`git status` без незакоммиченных изменений).
4. Если задача неоднозначна или не имеет критериев приёмки — **задаёт уточняющие вопросы пользователю** или предлагает декомпозицию на под-задачи в Linear.

**Правило:** агент не берёт две задачи одновременно. **Один Linear-issue (`PTR-XXX`) = один PR** — не закрывать несколько несвязанных задач одним PR; связанные задачи указывать через `Refs` (см. [`50-task-management.md`](50-task-management.md)). Одна активная задача → одна ветка → один PR.

## Шаг 1. Перевод задачи в работу

В Linear:

- Назначить задачу на себя (или соответствующего агента-исполнителя).
- Перевести статус в **In Progress**.
- Если в задаче нет критериев приёмки или test plan — добавить их комментарием перед началом работы.

## Политика ptr_coder: когда делегировать исполнение

**Исполнение через ptr_coder — норма**, а не исключение: им реализуются все задачи, где в репозитории нужно **менять исполняемые артефакты** — **код**, **конфигурации** (в том числе Docker, Compose, CI-манифесты, env-шаблоны), **скрипты**, **исправления багов**, **новые фичи**, **рефакторинг**, **инфраструктурные правки** в файлах репозитория и любая сопутствующая инженерная работа того же класса.

**Cursor ведёт сам (без ptr_coder как исполнителя кода)** только там, где нет такой реализации или она сознательно отложена:

- **Чистая документация** — тексты в markdown, комментарии в Linear, описания без изменения кода и конфигов в дереве (если в задаче по сути только документирование — ptr_coder не вызывается).
- **Архитектурные и продуктовые решения** — выбор подхода, границы системы, критерии приёмки, декомпозиция, споры по дизайну до фиксации плана; оформление решений в ADR/спеках **без** одновременной правки кода/конфигов (см. также границы в [`70-orchestration-ptr-coder.md`](70-orchestration-ptr-coder.md)).
- **Оркестрация ptr_coder** — подготовка handoff, запуск `python -m ptr_coder`, **независимая валидация** результатов, итерации, оформление коммитов и PR.

Как только по задаче принято решение «что именно менять в репо» и это попадает под первый абзац — **реализация идёт через ptr_coder** по handoff в `.ai/handoffs/` и процедуре из [`70-orchestration-ptr-coder.md`](70-orchestration-ptr-coder.md). Тривиальная правка одного файла не освобождает от ptr_coder, если это всё ещё код или конфиг.

## Шаг 2. Планирование

Для всего, кроме тривиальных **документных** правок (опечатка или форматирование одного markdown без изменения кода и конфигов), агент формулирует план:

- **Тип задачи** (`feature` / `bugfix` / `refactor` / `docs` / `infra`) — определяет акценты DoD и подходящую персону для делегирования (см. [`11-work-types.md`](11-work-types.md) и [`65-personas.md`](65-personas.md)).
- Цель и ожидаемый результат.
- Список файлов в скоупе (новые / изменяемые).
- Критерии приёмки (что считается «готово»).
- План тестирования.
- Известные риски и edge cases.

Если задача по разделу «Политика ptr_coder: когда делегировать исполнение» требует изменений кода или конфигов — план **обязательно** фиксируется в файле `.ai/handoffs/PTR-XXX-<slug>.md` (см. [`70-orchestration-ptr-coder.md`](70-orchestration-ptr-coder.md)) с явным указанием персоны **до** вызова исполнителя.

## Шаг 3. Создание ветки

```bash
git checkout -b PTR-XXX-<kebab-slug>
```

Подробности — в [`20-git-and-branching.md`](20-git-and-branching.md).

## Шаг 4. Реализация

- Изменения кода и конфигов по задаче выполняет **ptr_coder** по handoff (см. раздел «Политика ptr_coder: когда делегировать исполнение» выше и [`70-orchestration-ptr-coder.md`](70-orchestration-ptr-coder.md)); Cursor не пишет эти изменения «от руки» в обход ptr_coder, кроме случаев из второго абзаца той политики (документация, решения без правок дерева, оркестрация).
- Делать **малые, атомарные коммиты** в стиле Conventional Commits (см. [`30-commits-and-prs.md`](30-commits-and-prs.md)).
- На каждом значимом шаге — прогон тестов и линтеров локально.
- Не накапливать «всё в одном коммите в конце».
- Не менять файлы вне заявленного скоупа без явного основания (если нашёл побочную проблему — создать отдельную задачу в Linear).

## Шаг 5. Локальная валидация (pre-PR gate)

Перед `git push` агент **обязан** прогнать локально:

| Проверка | Команда (примерная — будет обновлена с появлением стека) |
|---|---|
| Линтер | `<lint-cmd>` |
| Форматер | `<format-cmd>` (или `--check`) |
| Типы | `<typecheck-cmd>` |
| Тесты | `<test-cmd>` |
| **Для UI:** Playwright/E2E | `<e2e-cmd>` |
| **Сервисы из `docker-compose.yml` изменены?** | См. **«Строгая проверка в Docker»** в [`40-code-quality.md`](40-code-quality.md): `docker compose --env-file .env.development config`, `build` / smoke для затронутых сервисов (детали в [`docs/tech/docker-dev.md`](../../docs/tech/docker-dev.md)) |

Если хотя бы одна проверка падает — **не пушить**, починить локально.

## Шаг 6. Push и открытие PR

```bash
git push -u origin HEAD
gh pr create --title "<type>(<scope>): PTR-XXX <description>" --body "$(cat <<'EOF'
... шаблон из .github/pull_request_template.md ...
EOF
)"
```

Заполнить **все секции** PR-шаблона. Проставить `Closes PTR-XXX` (если задача — единственная закрываемая) или `Refs PTR-XXX`.

## Шаг 7. Обновление Linear

- Перевести задачу в **In Review**.
- Оставить комментарий со ссылкой на PR.

## Шаг 8. Передача человеку для merge

Агент **никогда не мерджит свой PR** ([`90-forbidden.md`](90-forbidden.md)) — это процедурное правило, **не зависящее** от настроек branch protection (см. [ADR-0004](../../docs/adr/0004-relax-required-review.md)).

Действия после открытия PR:

- При комментариях ревьюера — ответить, при необходимости внести правки и запушить.
- При зелёном CI **информировать человека о готовности к merge** (комментарий в PR и/или Linear). Не мерджить.
- Если человек хочет посмотреть код перед merge — дождаться его реакции. Approving review технически не требуется, но если запрошен — учесть комментарии.

## Шаг 9. После merge

- Локально удалить feature-ветку: `git branch -d PTR-XXX-<slug>`.
- Удалённую ветку GitHub удаляет автоматически (включено `delete branch on merge`).
- Перевести задачу в Linear в **Done** (если это не сделано автоматически интеграцией Linear↔GitHub).

## Прерывание задачи

Если агент не может завершить задачу (внешний блокер, недостаток информации, найдены архитектурные противоречия):

1. **Не оставлять полусделанные коммиты в открытом PR** без пояснения.
2. Перевести задачу в Linear в статус, отражающий блокер (например, **Blocked**).
3. Оставить комментарий с описанием блокера и того, что уже сделано.
4. Если PR уже открыт — пометить его как **Draft** и описать в теле текущее состояние.
