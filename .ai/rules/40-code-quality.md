# 40 — Качество кода и Definition of Done

## Definition of Done (полная версия)

PR может быть смерджен **только если все** пункты ниже выполнены или явно помечены как «не применимо» в теле PR с обоснованием.

### Обязательные гейты

1. **Формат коммитов и PR.** Все коммиты соответствуют Conventional Commits с обязательным `scope` (см. [`30-commits-and-prs.md`](30-commits-and-prs.md)). Заголовок PR содержит `PTR-XXX`.
2. **Линтер.** Чистый прогон, ноль ошибок и ноль варнингов (либо явно whitelisted в конфиге).
3. **Форматер.** Код отформатирован каноничным форматером языка.
4. **Типы.** В типизированных языках — проверка типов (`tsc`, `mypy`, `pyright`, `go vet`, `cargo check` и т. п.) проходит без ошибок.
5. **Тесты.**
   - Новый функционал покрыт тестами.
   - Изменения в существующем коде сопровождаются обновлением соответствующих тестов.
   - Все тесты зелёные **локально** перед `git push`.
6. **UI / E2E (если PR затрагивает интерфейс).**
   - Обязательно добавлен или обновлён **Playwright** (или согласованный эквивалент) тест, подтверждающий, что код реализует ожидаемое поведение в **реальном браузере**.
   - Скриншоты/трейсы Playwright прикладываются в комментарии к PR при ручной проверке регрессий.
7. **CI зелёный.** Все обязательные status checks на стороне GitHub проходят.
8. **Документация.** Обновлены README/ADR/`.ai/rules/`/комментарии к публичному API при необходимости. Если PR меняет публичное поведение, документация **обязательна**.
9. **Human merge.** PR мерджит **человек** (`@enotbert` или текущий CODEOWNER). Approving review **технически не требуется** branch protection (см. [ADR-0004](../../docs/adr/0004-relax-required-review.md)), но **рекомендован** для PR типов `feat`, `fix`, `infra` и любых нетривиальных правок. Агент **никогда не мерджит свой PR** — это процедурное правило, см. [`90-forbidden.md`](90-forbidden.md).
10. **Docker Compose (строго).** Для любого PR, который меняет сервисы из корневого `docker-compose.yml` (backend, frontend, postgres) или их Dockerfiles/входные env — **при declare «готово к PR»** агент обязан прогнать **проверку через Docker** по [`docs/tech/docker-dev.md`](../../docs/tech/docker-dev.md): см. раздел «Строгая проверка в Docker» ниже. Хостовые команды (`pnpm`, `uv`) допускаются для скорости итерации, но **не заменяют** Compose-проверку для затронутых сервисов.

### Строгая проверка в Docker (локальный dev stack)

Корневой Compose описан в [`docs/tech/docker-dev.md`](../../docs/tech/docker-dev.md). Политика:

1. **Файл окружения для Compose:** в корне репозитория используется **`.env.development`** (в `.gitignore`, в git **не** коммитится). Шаблон — **`.env.development.example`** (в git). Если `.env.development` отсутствует — агент **создаёт** его копированием из примера (`cp` / аналог), **без** вывода содержимого в чат/PR (см. также [`80-security-and-secrets.md`](80-security-and-secrets.md)).
2. **Валидность env:** перед smoke-прогоном — `docker compose --env-file .env.development config` (exit **0**). Иначе исправить переменные или синтаксис в `.env.development` / шаблоне.
3. **Минимум перед push / «готово к ревью»** для затронутого сервиса:
   - `docker compose --env-file .env.development build <service>` — успешная сборка образа;
   - для **frontend** — доступность главной с хоста после `up` (например `scripts/smoke-frontend-docker.sh` или эквивалент из `docs/tech/docker-dev.md`);
   - для **backend** — задокументированная команда из `docs/tech/docker-dev.md` (например `run ... pytest` или health по HTTP), если PR трогает backend.
4. **Исключения** помечаются в теле PR («не применимо: …») с обоснованием (например нет Docker на машине агента — тогда **явный** запрос человеку на прогон; по умолчанию считаем, что Docker доступен).

Смысл: «локально работает у разработчика вне контейнера» **не достаточно** для закрытия DoD по compose-сервисам — источник истины для интеграции — **Compose**.

### Дополнительные требования

- **Никаких отключений проверок** (`eslint-disable`, `# noqa`, `// @ts-ignore`, `# type: ignore` и аналогов) без обоснования в комментарии **прямо рядом**.
- **Никаких `--no-verify`** при коммите/push.
- **Никаких закомментированных блоков кода.** Если код может понадобиться — он сохраняется в git-истории, а не в комментариях.
- **Никаких TODO без задачи.** Любой `TODO`/`FIXME` сопровождается ссылкой на задачу в Linear: `// TODO(PTR-123): ...`.

## Стиль кода

- **Используй каноничные форматеры** для каждого языка. Конкретные конфиги добавляются ADR при появлении языка в репо.
- **Не вводи новый стиль** в существующий пакет без ADR.
- **Имена** — самодокументирующиеся; короткие сокращения только для общеупотребимых (`id`, `db`, `url`, `ui`).

## Комментарии

- Комментарий объясняет **намерение, ограничение или нетривиальный выбор**, а не повторяет то, что и так видно в коде.
- Запрещены комментарии вида `// import the module`, `// increment counter`, `# return result`.
- Комментарии — на английском (см. языковую политику в [`AGENTS.md`](../../AGENTS.md#2-языковая-политика)).
- Для сложных алгоритмов — короткое описание идеи перед функцией; при необходимости — ссылка на статью/ADR.

## Тесты

- **Юнит-тесты** — для логики, не зависящей от внешних систем.
- **Интеграционные** — для границ модулей и I/O.
- **E2E (Playwright)** — для UI-флоу, как описано выше.
- Тесты должны быть **детерминированными**. Никаких случайных таймаутов и `sleep` для синхронизации.
- Не использовать в тестах продакшен-секреты или реальные внешние API (только моки/sandbox).

## Зависимости

Изменение зависимостей (`package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml`, lockfiles) — действие из чёрного списка для агента (см. [`90-forbidden.md`](90-forbidden.md)). Требует:

1. Явного запроса пользователя.
2. Отдельного PR с типом `build` или `chore`.
3. Обоснования: зачем, какая альтернатива, какой license.

## Производительность и безопасность

- Не оптимизируй преждевременно. Бенчмарк → решение → бенчмарк.
- Никогда не добавляй пользовательский ввод напрямую в SQL/shell/HTML без санитизации.
- Логи не должны содержать секретов или PII.
