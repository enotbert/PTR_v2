# 11 — Флоу по типам работ

> Базовый цикл — в [`10-workflow.md`](10-workflow.md). Здесь — отличия и акценты для разных типов задач.
> Тип определяется на этапе **планирования** (Шаг 2 базового цикла) и фиксируется в Linear-метке и в Conventional Commit `<type>` коммитов/PR.

## Соответствие типов

| Тип задачи | CC `<type>` | Типичная Linear-метка |
|---|---|---|
| Новая фича | `feat` | `feature` |
| Исправление бага | `fix` | `bug` |
| Рефакторинг | `refactor` | `tech-debt`, `refactor` |
| Документация и правила | `docs` | `docs` |
| Инфраструктура / CI / релизы | `ci`, `build`, `chore` | `infra`, `ci` |

> Метки в Linear на старте отсутствуют. Создаются по мере появления соответствующих задач (через `create_issue_label` MCP), фиксируется в ADR при первом использовании.

---

## Feature

**Цель:** добавить новую функциональность, видимую пользователю или другому компоненту.

### Особенности планирования

- Обязательно сформулировать **критерии приёмки** в терминах поведения, а не реализации.
- Если фича затрагивает несколько пакетов/сервисов — декомпозировать на под-задачи **до** начала работы (`feat(api):` + `feat(web):` отдельными PR, привязанными к одной parent-задаче).
- Любая UI-фича **обязательно** включает Playwright/E2E сценарий (см. [`40-code-quality.md`](40-code-quality.md)).

### Дополнительные пункты DoD

- [ ] Добавлены/обновлены unit-тесты для новой логики
- [ ] Добавлен интеграционный тест на сценарий end-to-end (если фича пересекает границы модулей)
- [ ] Для UI: Playwright покрывает основной user flow и хотя бы один edge case
- [ ] Документация обновлена (README пакета или ADR при существенных архитектурных изменениях)
- [ ] Telemetry/logging добавлены, если фича пользовательская и важна для observability

### Типичная персона делегирования

`backend` или `frontend` (см. [`65-personas.md`](65-personas.md)) — в зависимости от слоя. Для отдельного дизайн-скоупа (спеки, токены, статические ассеты, моки) — **`designer`**, затем при необходимости **`frontend`** для имплементации в коде приложения.

---

## Bugfix

**Цель:** устранить расхождение фактического поведения с ожидаемым.

### Особенности планирования

- **Сначала тест, потом фикс.** Воспроизвести баг тестом (упавшим), затем правкой добиться зелёного.
- В описании задачи Linear обязательно: **steps to reproduce**, **expected**, **actual**, **observed in environment** (версия/окружение).
- Если корневая причина шире проявления — оформить отдельную задачу `tech-debt` для устранения первопричины, а текущим PR закрыть симптом (это явно зафиксировать в теле PR).

### Дополнительные пункты DoD

- [ ] Добавлен **регрессионный тест**, который падает без фикса и проходит с ним
- [ ] В описании PR указана корневая причина (или отсылка к follow-up задаче, если сейчас закрывается только симптом)
- [ ] Проверены **смежные сценарии**: не сломал ли фикс соседнее поведение (явно перечислить, что прогнал)
- [ ] Для UI-багов: Playwright-тест на конкретный воспроизведённый сценарий

### Типичная персона делегирования

`backend`/`frontend` под слой бага; для сложных диагностических багов — сначала Cursor разбирается, потом делегирует фикс.

---

## Refactor

**Цель:** изменить **внутреннюю** структуру кода без изменения внешнего поведения.

### Особенности планирования

- **Поведение не меняется.** Если меняется — это `feat`, `fix` или `perf`, не `refactor`.
- Перед рефакторингом убедиться, что **существующие тесты адекватно покрывают** изменяемую область. Если нет — сначала добавить тесты (отдельным PR с типом `test`), потом рефакторить.
- Изменения должны быть **атомарными** и **обратимыми**: один логический шаг — один PR.

### Дополнительные пункты DoD

- [ ] Все существующие тесты проходят **без модификаций** (если тесты пришлось менять — это сигнал, что поведение всё-таки изменилось)
- [ ] Тестовое покрытие не уменьшилось
- [ ] Бенчмарки (если есть) не показали регрессии
- [ ] В теле PR явно указано, что **поведение не изменилось** и какие гарантии этого

### Типичная персона делегирования

Для механического рефакторинга (переименования, перемещения) — `backend`/`frontend`. Для архитектурного — Cursor сам, без делегирования.

## Docs

**Цель:** улучшить документацию, правила, ADR без изменения исполняемого кода.

### Особенности планирования

- **Никаких изменений кода в этом же PR.** Если правка документации сопровождается правкой кода — это два отдельных PR.
- Изменения в `AGENTS.md`, `.ai/rules/`, `docs/adr/` — **только с явного запроса пользователя** (см. [`90-forbidden.md`](90-forbidden.md)).
- Перевод/локализация — отдельный тип, не смешивать с правкой содержания.

### Дополнительные пункты DoD

- [ ] Markdown рендерится без артефактов (проверить в GitHub UI или локальным рендером)
- [ ] Все внутренние ссылки резолвятся (нет «битых» относительных путей)
- [ ] Терминология консистентна с остальной документацией
- [ ] Если затронуты правила (`.ai/rules/`) — обновлены перекрёстные ссылки в смежных файлах и в `AGENTS.md`
- [ ] Если документ ссылается на код — фрагменты актуальны

### Типичная персона делегирования

`docs` (см. [`65-personas.md`](65-personas.md)). Для системных правок правил — Cursor сам.

## Infra

**Цель:** изменить инфраструктуру: CI, сборку, конфиги репозитория, контейнеризацию, deployment, IaC.

### Особенности планирования

- **Высокий риск.** Ошибка в CI или branch protection может заблокировать всю работу команды/агентов.
- Любое изменение `.github/`, branch protection, `CODEOWNERS`, lockfiles, версий рантайма — **только с явного запроса пользователя** (см. [`90-forbidden.md`](90-forbidden.md)).
- Для нетривиальных изменений CI — сначала **dry run** в feature-ветке, потом merge.
- Не смешивать infra-PR с продуктовыми изменениями.

### Дополнительные пункты DoD

- [ ] Изменение **успешно прогналось в CI** на самой feature-ветке
- [ ] Документировано в `README` соответствующего пакета или в ADR (для значимых решений)
- [ ] Откат-план описан в теле PR: что делать, если изменение сломает main
- [ ] Если изменение касается секретов/токенов — координация с владельцем (`@enotbert`), вне репо
- [ ] Для миграций БД и подобных необратимых операций — отдельный ADR с обоснованием

### Типичная персона делегирования

`infra` (см. [`65-personas.md`](65-personas.md)) **только** для безопасных операций (smoke checks, форматирование workflow, миграция action versions). Всё остальное — Cursor под прямым контролем человека.

## Когда тип задачи смешанный

Если задача объективно требует одновременно нескольких типов (например, новая фича + добавление инфраструктуры под неё):

1. **Декомпозировать на отдельные PR** — это всегда первый выбор.
2. Если декомпозиция нецелесообразна — выбрать **доминирующий** тип для CC (`feat` обычно превалирует над `chore`/`ci`/`docs`).
3. В теле PR явно перечислить вторичные изменения.
4. Размер PR при этом обычно превышает 400 строк — обязательно обоснование (см. [`30-commits-and-prs.md`](30-commits-and-prs.md) → Размер PR).

## Соответствие типам в DoD-чеклисте PR

[Шаблон PR](../../.github/pull_request_template.md) содержит общий чеклист. При работе над задачей конкретного типа агент **дополнительно** проверяет соответствующие пункты выше и упоминает их в теле PR (раздел «Дополнительно»).
