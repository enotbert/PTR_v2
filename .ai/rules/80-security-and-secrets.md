# 80 — Безопасность и секреты

## Базовая политика

В репозитории **никогда** не хранятся реальные секреты, ключи, токены, пароли или иные чувствительные данные.

Что разрешено в репо:

- Файл `.env.example` — шаблон с **именами** переменных и пустыми значениями (или demo-плейсхолдерами).
- Документация о том, **где** взять реальные секреты (без самих секретов).

Что запрещено в репо (всегда, без исключений):

- Файлы `.env`, `.env.local`, `.env.production` и любые другие реальные `.env*` (кроме `.env.example`).
- Файлы `*.key`, `*.pem`, `*.p12`, `*.pfx`, `id_rsa`, `id_ed25519`, `*.crt` (для приватных ключей; публичные сертификаты — обсуждается отдельно).
- `credentials.json`, `service-account*.json`, `*-secret.json`.
- Файлы дампов БД с реальными данными.
- Hardcoded значения в коде (`API_KEY = "sk-..."`).

Эти паттерны включены в корневой `.gitignore`.

## Что агент может делать с секретами

| Действие | Можно? |
|---|---|
| Читать `.env.example` | ✅ |
| Создавать/обновлять `.env.example` | ✅ (с обновлением документации) |
| Читать `.env`, `*.key`, `credentials.*` | ❌ Никогда |
| Выводить содержимое `.env`/`*.key` в логи, чат, PR | ❌ Никогда |
| Передавать содержимое `.env` другим агентам или внешним системам | ❌ Никогда |
| Добавлять переменную в `.env.example` (с пустым/demo значением) | ✅ |
| Добавлять переменную в реальный `.env` | ❌ Это делает человек |

Если агенту в процессе работы потребовался доступ к секрету — он **останавливается** и просит человека либо добавить переменную в окружение, либо предоставить тестовый sandbox-ключ.

## Локальная разработка

- `.env` живёт **только** в рабочем дереве разработчика, отсутствует в `git`.
- Каждое приложение/сервис в монорепе имеет свой `.env.example` рядом со своим конфигом.
- Корневой `.env.example` — для глобальных переменных, общих по всей монорепе.

## CI / GitHub Actions

- Все секреты для CI хранятся в **GitHub Actions Secrets** (settings → secrets and variables → actions).
- Не использовать `echo $SECRET` в шагах (значение появится в логах).
- Использовать встроенные механизмы: `secrets.MY_SECRET`, маскирование, environments с required reviewers для деплоев.
- Внешние интеграции (Linear, любые API) — токены через GH Secrets, никогда в коммитах.

## Сканирование секретов

### Что сейчас работает

- ✅ **GitHub Secret Scanning** — включено по умолчанию для публичного репо. Алерты в Security → Secret scanning alerts.
- ✅ **Push Protection** — включено. GitHub блокирует push, если в нём обнаружен паттерн известного провайдера (AWS keys, GitHub tokens, Stripe и др.).

### Что отложено

- **`gitleaks` в CI** — отложено до появления первого пакета/сервиса, работающего с внешними API/секретами. На текущем этапе встроенных средств GitHub достаточно.
- **Pre-commit hook `gitleaks protect`** — будет настроен одновременно с базой git-хуков (см. соответствующий пункт в [`AGENTS.md` §9](../../AGENTS.md#9-открытые-вопросы)).

### Когда подключаем gitleaks

Триггеры для подключения (любой из):

1. Первый PR, добавляющий зависимость от внешнего сервиса с API-ключами.
2. Первый ложноположительный пропуск GitHub'ом (gitleaks ловит паттерны точнее, имеет кастомизируемые правила).
3. Появление приватных репо/форков, на которые GitHub Secret Scanning платный.

При подключении — оформить ADR с обоснованием и конфигом (`.gitleaks.toml`).

## Что делать, если секрет случайно попал в репо

1. **Немедленно** считать его скомпрометированным: ротировать в источнике (провайдер API, БД и т. п.).
2. Удалить из истории: `git filter-repo` или BFG (это операция требует force-push на `main` — **только с явного разрешения человека**, агент не выполняет сам).
3. Добавить паттерн в `.gitignore` и/или сканер, чтобы не повторилось.
4. Создать ADR-инцидент в `docs/adr/`: что произошло, что сделано, что изменено в процессе.

## Зависимости и supply chain

- Не добавлять зависимости из неподтверждённых источников.
- Перед добавлением — проверить актуальные CVE и активность поддержки.
- Не отключать автоматические security-updates (Dependabot — будет настроен, см. [§9](../../AGENTS.md#9-открытые-вопросы-требуют-решения)).
