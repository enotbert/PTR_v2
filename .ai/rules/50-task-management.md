# 50 — Управление задачами (Linear)

## Источник истины

**Linear** — единственный официальный трекер задач. Префикс команды — `PTR`.

- Все ID задач имеют вид `PTR-XXX`.
- Каждое нетривиальное изменение должно быть привязано к существующей задаче.
- Если задачи нет — её нужно создать **до** начала работы.

## Жизненный цикл задачи

| Статус | Когда выставлять | Кто выставляет |
|---|---|---|
| `Backlog` / `Todo` | Задача создана, ещё не взята | Человек, агент-оркестратор |
| `In Progress` | Агент начал работать (создал ветку) | Агент-исполнитель |
| `In Review` | PR открыт и переведён в Ready | Агент-исполнитель (после открытия PR) |
| `Blocked` | Найден внешний блокер; работа приостановлена | Агент-исполнитель (с комментарием-причиной) |
| `Done` | PR смерджен | Автоматически (Linear↔GitHub) или человек |
| `Cancelled` | Работа отменена | Только человек |

## Что агент делает в Linear

Разрешено (см. [`AGENTS.md` §3 — Роли](../../AGENTS.md#3-роли-агентов) и [Round 2 решения]):

- ✅ Менять статус задачи между `Todo`/`In Progress`/`In Review`/`Blocked` для задач, над которыми работает агент.
- ✅ Оставлять комментарии: ссылки на PR, краткие отчёты о статусе, найденные блокеры.
- ✅ **Создавать новые задачи и под-задачи** для:
  - декомпозиции взятой задачи на под-шаги (под-задачи под `PTR-XXX`),
  - follow-up задач, обнаруженных в ходе работы (новый top-level `PTR-YYY`),
  - багов/долгов, найденных мимоходом (новая задача, без правок «по пути»).
- ✅ Устанавливать связи между задачами (parent/child, blocks/blocked-by, related).

Запрещено без явного разрешения человека:

- ❌ Менять статус на `Done` или `Cancelled` (это делает человек или интеграция при merge).
- ❌ Удалять задачи.
- ❌ Менять приоритет, оценки, исполнителя на других людей/команды.
- ❌ Закрывать или менять чужие задачи (которые агент не брал).

## Создание задачи агентом

При создании задачи агент **обязан** указать:

- **Title** — краткое описание (≤ 80 символов, в императиве).
- **Description** —
  - Контекст / проблема.
  - Критерии приёмки (что считается «готово»).
  - План тестирования (если возможен).
  - Если задача — follow-up другой работы: ссылку на исходный PR/задачу.
- **Labels** (если применимо) — `agent-created`, `tech-debt`, `bug`, `infra`, …
- **Parent** — если это под-задача.

## Один issue = один PR

- **Норма:** на задачу `PTR-XXX` открывается **ровно один** PR, который её закрывает. Отдельная задача в Linear — отдельная ветка и отдельный PR (имя ветки — [`20-git-and-branching.md`](20-git-and-branching.md)).
- **Не делать:** смешивать в одном PR закрытие нескольких **независимых** задач ради скорости или общего ревью.
- **Связанные задачи**, которые этот PR не закрывает полностью, указывать через `Refs PTR-YYY` (без второго `Closes` в том же PR). Если объём расползся — новая задача `PTR-YYY` и новый PR.
- **Исключение:** автоматические release-PR (см. [`30-commits-and-prs.md`](30-commits-and-prs.md) — нет привязки к одному `PTR-XXX`).

## Связь с GitHub

- В **имени ветки** ID присутствует обязательно: `PTR-XXX-<slug>`.
- В **заголовке PR** ID обязателен: `<type>(<scope>): PTR-XXX <description>`.
- В **теле PR** магическая фраза для авто-закрытия:
  - `Closes PTR-XXX` — **одна** задача, которую этот PR полностью закрывает (**один issue = один PR**).
  - `Refs PTR-YYY` — для связи без закрытия (follow-up, контекст, блокеры).

> Linear автоматически распознаёт ID `PTR-XXX` в commit messages, branch names и PR descriptions при настроенной интеграции GitHub↔Linear. Магические слова (`Closes`, `Fixes`, `Resolves`) переводят задачу в `Done` при merge.

## Если задача неоднозначна

Перед началом работы:

1. Прочитать описание полностью, включая комментарии.
2. Если критериев приёмки нет — **сформулировать их и оставить комментарием в задаче**, дождаться подтверждения от человека.
3. Если задача содержит несколько независимых направлений — предложить декомпозицию на под-задачи и **дождаться согласия**.
4. Если задача требует архитектурного решения — поднять ADR (см. [`docs/adr/`](../../docs/adr/)) до начала кодинга.

## Если задача распухает в процессе

- Не расширять скоуп текущего PR.
- Создать отдельную задачу `PTR-YYY` с отметкой `related to PTR-XXX`.
- Добавить ссылку в комментарий исходной задачи.
